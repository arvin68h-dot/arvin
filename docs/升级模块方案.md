# 升级模块 + 自动文档更新 - 实施方案

## 目标

1. **数据库升级模块** — 支持未来 schema 变更自动迁移
2. **变更时自动更新操作手册** — 代码变更自动同步文档版本号和章节内容

---

## 一、数据库升级模块 (Storage Migration)

### 1.1 当前状态

- `packages/storage/src/index.ts` 已有 `schema_migrations` 表（空壳）
- `SCHEMA_VERSION = 1` 硬编码，无实际迁移逻辑
- 新数据库直接跑完整 DDL，不会执行增量升级

### 1.2 方案：Migration 系统

```typescript
// packages/storage/src/migrations/index.ts
// packages/storage/src/migrations/001_initial.ts
// packages/storage/src/migrations/002_add_column.ts  ...
```

**核心设计：**
- 每个迁移文件导出 `{ version: number; up(sql: string): void; down?(sql: string): void }`
- `initializeDatabase()` 检查 `schema_migrations` 表，执行未运行的迁移
- 升级流程：`currentVersion → SCHEMA_VERSION`，按序执行 `up()`

**迁移文件结构：**
```typescript
// migrations/002_add_task_group.ts
export default {
  version: 2,
  up: (sql: string[]) => [
    'ALTER TABLE tasks ADD COLUMN group_id TEXT',
    "CREATE INDEX IF NOT EXISTS idx_tasks_group ON tasks(group_id)",
  ],
  down: (sql: string[]) => [
    'ALTER TABLE tasks DROP COLUMN group_id',
  ],
};
```

### 1.3 升级流程

```
新版本发布 → SCHEMA_VERSION 递增
           → 新增 migrations/003_xxx.ts
           → initializeDatabase() 自动检测并执行增量迁移
           → 记录到 schema_migrations 表
```

### 1.4 CLI 命令

```bash
# 查看当前版本
codeengine db version

# 查看可升级的迁移
codeengine db migrations list

# 手动触发升级
codeengine db migrate
```

## 二、自动文档更新

### 2.1 触发机制

每次 Git commit 后自动运行文档同步脚本，检测变更的文件并更新操作手册。

### 2.2 文档同步规则

| 变更位置 | 自动更新内容 |
|----------|-------------|
| `packages/storage/index.ts` 变更 | 更新版本号、最后更新日期、存储相关章节（数据库表结构、字段说明） |
| `packages/storage/migrations/` 新增 | 追加"数据库升级"章节 |
| CLI 命令变更 (`cli/src/commands/`) | 更新第5节 CLI 命令参考 |
| 配置结构变更 (`packages/core/src/config/`) | 更新第4节 配置管理 |
| 工具定义变更 (`packages/tool/`) | 更新第6节 工具系统 |
| 引擎定义变更 (`packages/engine/`) | 更新第7节 语言引擎 |

### 2.3 实现方案

**不引入新依赖，用纯 Node.js 脚本实现：**

```
scripts/docs-sync.ts          # 主入口：检测变更 + 生成更新内容
scripts/doc-parser.ts         # 解析操作手册.md，提取章节
scripts/doc-updater.ts        # 按规则更新章节内容
```

**工作方式：**
1. `git diff HEAD~1 --name-only` 获取变更文件列表
2. 根据变更文件类型，调用对应更新器
3. 更新版本号 (`0.1.0 → 0.1.1`)
4. 更新"最后更新"日期
5. 重新生成对应章节内容（从代码注释/类型定义提取）
6. 写入 `docs/操作手册.md`
7. 如果有变更，自动 `git add + git commit`（可配置开关）

### 2.4 文档版本号策略

使用语义化版本，由变更类型自动 bump：
- 新功能/新表/新字段 → 补丁版本 (`0.1.0 → 0.1.1`)
- 破坏性变更 (表删除/字段改名) → 小版本 (`0.1.0 → 0.2.0`)

## 三、实施步骤

### Step 1: 数据库升级模块
1. 在 `packages/storage/src/` 下新建 `migrations/` 目录
2. 编写 `001_initial.ts`（将当前完整 DDL 拆分为初始迁移）
3. 修改 `initializeDatabase()` 使用迁移系统
4. 编写单元测试
5. 在 CLI 中添加 `db version` / `db migrate` 命令
6. 更新操作手册相关章节

### Step 2: 自动文档更新
1. 编写 `scripts/doc-parser.ts` — 解析操作手册.md 的章节结构
2. 编写 `scripts/doc-updater.ts` — 根据代码变更更新对应章节
3. 编写 `scripts/docs-sync.ts` — 主入口（cli command 触发）
4. 编写单元测试
5. 更新操作手册，添加"文档与版本"章节

### Step 3: 集成 CI/CD
1. `pr-check.yml` 中集成 `pnpm run docs:sync --check`
2. 验证文档版本与代码版本一致

## 四、风险评估

| 风险 | 影响 | 缓解 |
|------|------|------|
| 迁移失败导致数据丢失 | 高 | down() 回滚 + 自动备份 |
| 文档同步冲突 | 中 | 仅追加/更新特定章节，不覆盖全文 |
| Git auto-commit 干扰 | 低 | 可配置关闭，`--dry-run` 模式 |
